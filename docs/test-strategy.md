# 测试策略说明

## 1. 目标

本项目用于演示测试工程师如何围绕零售 SaaS 核心业务链路设计自动化回归用例。重点不是追求复杂框架，而是体现：

- 用例设计是否贴近真实业务风险
- 断言是否能覆盖金额、状态、库存、积分、权限等关键结果
- 测试数据是否可复用
- 回归链路是否能快速验证主流程

## 2. 测试范围

| 类型 | 覆盖内容 |
| --- | --- |
| 功能测试 | 登录、商品上下架、创建订单、支付、退款 |
| 接口测试 | 模拟接口请求和响应断言 |
| 数据一致性 | 商品多端展示、订单金额、库存、会员积分 |
| 权限测试 | 不同角色访问报表和敏感数据 |
| AI 应用测试 | AI 回复与后台报表数据一致性、无权限兜底 |
| 回归测试 | 商品到 AI 经营分析的完整主链路 |

## 3. 核心断言

- 登录成功后 token、角色、门店 ID 不为空
- 商品下架后掌柜 APP 和小程序均不展示
- 下单金额 = 商品单价 * 数量 - 优惠金额
- 支付成功后订单状态变为 PAID
- 重复支付回调不重复扣库存
- 支付后会员积分增加，退款后积分回滚
- 店员不能访问经营报表
- AI 助手输出的销售额和订单数与报表一致

## 4. 风险点

| 风险点 | 自动化验证方式 |
| --- | --- |
| 支付回调重复导致重复入账 | 使用相同 transaction_id 重复回调，验证幂等 |
| 商品多端展示不一致 | 同时查询掌柜 APP、收银 APP、小程序商品列表 |
| 优惠金额计算错误 | 校验 totalAmount、couponAmount、payAmount |
| 库存超卖或回补异常 | 支付扣减库存，退款恢复库存 |
| 权限越权 | 店员访问报表应返回 PERM_001 |
| AI 输出编造数据 | AI 回复必须带 evidence，并与报表接口一致 |

## 5. 后续可扩展方向

- 接入真实测试环境接口
- 使用 requests/httpx 封装 API client
- 增加 Allure 或 pytest-html 报告
- 增加 Jenkins/GitHub Actions 定时回归
- 增加数据库校验，例如 MySQL 查询订单、支付流水、库存表
- 将 Apifox 导出的接口数据转为 pytest 用例
